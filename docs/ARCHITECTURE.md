# Architecture: Mini System Monitor (MiSysMon)

本ドキュメントは、Windows環境向けの軽量システムモニターアプリケーション「Mini System Monitor (MiSysMon)」の設計構造、技術スタック、ディレクトリ構造の意図、およびデータフローについて説明します。

---

## 1. システムの概要と目的

### 概要
Mini System Monitor (MiSysMon) は、画面上に常駐する横長スリム型（1100px x 32px）の軽量システムモニターです。

### 目的
- **最小限のフットプリント**: 低スペックなPCやゲーム実行中などの環境を阻害しないよう、メモリ使用量とCPU使用率を極限まで低く保ちます。
- **高視認性**: 常に最前面に表示し、CPU使用率・温度、メモリ使用率、ネットワークトラフィック、ディスク使用量とIO、現在日時をコンパクトかつ一目で確認できるようにします。
- **誤操作防止と利便性**: 枠なし・透過のウィンドウでありながら、ドラッグによるスムーズな移動、次回起動時のウィンドウ位置の復元、多重起動防止などを提供します。

---

## 2. 技術スタック

本プロジェクトは以下の技術・ライブラリで構成されています。

| レイヤー / ライブラリ | 役割 | 理由・特徴 |
| :--- | :--- | :--- |
| **Rust** (Edition 2021) | 開発言語 | 高いメモリ安全性、ゼロコスト抽象化、低レイテンシ、最小限のリソースフットプリント。 |
| **egui / eframe** (0.35.0) | GUIフレームワーク | Immediate Mode (即時モード) の軽量GUI。GPU/CPU描画が効率的で、スリムなUIの実装に適している。 |
| **sysinfo** (0.38.0) | システムメトリクス取得 | OS非依存のインターフェースでCPU、メモリ、ネットワーク、ディスク、温度情報を取得。 |
| **chrono** (0.4.38) | 日付・時間 | ローカル日時の取得およびフォーマット。 |
| **serde / serde_json** | シリアライズ | ウィンドウ位置などの設定情報をシリアライズし、永続化ストレージへ保存。 |
| **common_lib** (ローカルクレート) | 共有ライブラリ | 親ディレクトリに配置された共通ライブラリ。多重起動防止用のミューテックス取得、バイト表記のフォーマット (`format_bytes`) を利用。 |

## 3. アーキテクチャとディレクトリ構造の意図

### ディレクトリ構造

```
MiSysMon/
├── .agents/            # AIエージェント用の指示書とルール
│   ├── AGENTS.md       # 自動ドキュメンテーションと品質管理ルール
│   └── INSTRUCTIONS.md # AIがコード変更時に遵守すべきコーディングルール
├── .github/            # GitHub Actions ワークフロー設定
├── docs/               # プロジェクト仕様・設計書などの各種ドキュメント
│   ├── ARCHITECTURE.md # 本ファイル (設計・アーキテクチャ)
│   ├── DEVELOPMENT.md  # 開発手順書
│   ├── DIAGRAM.md      # システム図 (Mermaid)
│   ├── FOOTPRINTS.md   # バイナリサイズとメモリフットプリントの記録
│   ├── RELEASE_FLOW.md # リリース手順書
│   ├── SPEC.md         # 機能および外観の仕様書
│   └── TEST_REPORT.md  # テスト結果報告書
├── scripts/            # ビルドおよび配信用スクリプト
├── src/                # ソースコード
│   └── main.rs         # アプリケーションの全ロジック (エントリーポイント, UI, メトリクス更新)
├── Cargo.toml          # Rust依存パッケージ設定
└── TODO.md             # 進捗管理および今後のタスク
```

### 構造の意図
- **単一エントリーポイントによる極限のシンプル化**:
  アプリケーション規模が小さいため、`src/main.rs` の単一ファイルに全てのロジック（UI、システムメトリクス取得、設定永続化、メインループ）を収めています。これによりコードのオーバーヘッドを減らし、コンパイラによる最適化（LTO）が最も効きやすい構成にしています。
- **共有ロジックの分離 (`common_lib`)**:
  プラットフォーム特有のAPIやフォーマット処理といった共通で使える低レベル処理は、隣接する共有ライブラリ `common_lib` に分離することで、MiSysMonの本体コードの肥大化を防ぎます。
- **厳密なリリース最適化設定 (`Cargo.toml`)**:
  `opt-level = 'z'`（サイズ最適化）、`strip = true`（シンボル削除）、`panic = 'abort'`（巻き戻し排除）など、バイナリサイズ削減に特化したプロファイルを設定しています。

## 4. データフローとモジュール間連携

### A. 起動と初期化フロー
1. **多重起動チェック**:
   `common_lib::desktop::acquire_single_instance` を通じて、システムに特定の名前付きミューテックスが存在するか確認します。すでに起動している場合は、即座に正常終了します。
2. **設定データのロードと表示座標検証**:
   `eframe` が提供する永続ストレージから、前回の終了時に保存されたウィンドウ位置情報（`Config`）を JSON デシリアライズして復元します。
   Windows環境の場合、復元された座標がいずれかの有効なモニターの表示範囲内にあるかを Win32 API の `MonitorFromPoint` を用いて検証します。どの画面の表示範囲にも含まれない（モニター切断や解像度変更による画面外化）場合は、座標の復元をキャンセルし、デフォルト位置（画面中央）にフォールバックします。
3. **ウィンドウの作成**:
   枠なし (`decorations(false)`)、透過背景 (`transparent(true)`)、常に最前面 (`always_on_top`) のオプションでウィンドウを立ち上げ、検証された位置に配置します。

### B. イベントループと描画・更新データフロー

```
 ┌────────────────────────────────────────────────────────┐
 │                      eframe/egui                       │
 │                   (Immediate Mode)                     │
 └──────────────────────────┬─────────────────────────────┘
                            │ (1秒ごとに呼び出し)
                            ▼
                  [ update_stats() ]
                            │
         ┌──────────────────┼──────────────────┐
         ▼                  ▼                  ▼
   sysinfo::System   sysinfo::Networks   sysinfo::Disks
   (CPU, Memory)        (Traffic)         (Disk Space)
         │                  │                  │
         └──────────────────┼──────────────────┘
                            ▼
              [ 差分計算 & データの格納 ]
               - CPU使用率 / 温度
               - メモリ使用率
               - ネットワーク速度(上り/下り)  *初期起動時は累積値を baseline に設定
               - ディスクIO速度 / 使用量
                            │
                            ▼
                   [ ui() 描画ループ ]
        - 各項目の固定幅ラベルに整形して描画
        - ドラッグ移動検知 ──> OSにドラッグを委譲
        - 時計表示 (chrono) 取得・更新
        - 終了ボタン (×) ──> ViewportCommand::Close
```

### C. 診断ログとログローテーション
- アプリケーション初期化時 (`SystemMonitor::new`) に、検出された温度センサー情報を `sensors_debug.log` に出力して診断に備えます。
- 起動するたびに、過去の診断ログを `sensors_debug.log.1`, `.2`, `.3` へと最大3世代ローテーションさせて、ディスク消費量を一定に抑えます。
- 検出センサー数が0個の場合、ログ内に管理者権限での起動を促すアドバイスを自動追記します。

1. **データ取得の最適化**:
   毎フレームごとの無駄なCPU消費を避けるため、1秒が経過するまでは `sysinfo` のリフレッシュを行わず、画面の再描画も最小限に抑えます。また、`sysinfo` は全体スキャン (`refresh_all`) を行わず、更新が必要な部分（CPU、メモリ、プロセスIO、ネットワーク）のみを対象として更新します。
2. **ウィンドウ位置の動的追跡**:
   ドラッグで移動されたウィンドウの座標は、eguiの入力ビューポート情報（`outer_rect`）から毎フレーム監視され、メモリ上の設定（`SystemMonitor::config.pos`）に常に最新の座標が格納されます。
3. **安全な終了と永続化**:
   ユーザーが「×」ボタンを押すと、閉じるコマンドが送信され、`eframe` が終了するタイミングで `save` メソッドが呼び出され、最終座標をシリアライズしてディスクに保存します。
