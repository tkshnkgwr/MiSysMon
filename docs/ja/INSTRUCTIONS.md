# AI Instructions: Coding Standards & Guidelines

本プロジェクトにおいて、AIエージェント（大賢者）がソースコードの変更、修正、追加を行う際に遵守すべきコーディング規約および出力フォーマットに関する指示書です。


---

## 1. 命名規則 (Naming Conventions)

Rustの標準的なコーディングスタイル (RFC 430) に厳密に従います。

- **型・構造体・列挙型・トレイト名**:
  `UpperCamelCase` (PascalCase) を使用します。
  - *例*: `SystemMonitor`, `Config`, `NetworkStats`
- **関数・メソッド・変数・マクロ・モジュール名**:
  `snake_case` を使用します。
  - *例*: `update_stats`, `cpu_usage`, `net_down`
- **定数・スタティック変数**:
  `SCREAMING_SNAKE_CASE` を使用します。
  - *例*: `const UPDATE_INTERVAL: Duration = ...`
- **ファイル名**:
  - ソースコードおよびスクリプト: `snake_case.rs`
  - ドキュメントファイル（Markdown等）: **大文字スネークケース (`SCREAMING_SNAKE_CASE.md`)**
    - *例*: `SPEC.md`, `ARCHITECTURE.md`, `INSTRUCTIONS.md`, `TODO.md`, `README_JA.md`

## 2. エラーハンドリング方針 (Error Handling)

Rustの安全性を損なわない堅牢なコードを記述します。

- **`panic!` や `unwrap()` の原則禁止**:
  ライブラリのパニックや意図しないクラッシュを防ぐため、`.unwrap()` や `.expect()` はテストコード以外では極力使用しません。
- **戻り値によるエラー伝搬**:
  失敗する可能性のある操作には `Result<T, E>` や `Option<T>` を返し、呼び出し元で適切に処理します。
- **安全なデフォルト値・フォールバックの適用**:
  - ハードウェア情報（温度など）の取得に失敗した場合や、未対応OSでの実行時には、デフォルト値（例: `0.0`）を返却するか、安全にフォールバック（例: センサーが存在しない場合は最高温度を採用）するロジックを実装します。
  - *例*: `unwrap_or(0.0)`, `unwrap_or_default()` などを利用。
- **ファイルI/Oや設定読み込み時のエラーハンドリング**:
  `Result` を受け取った際は、`if let Ok(...)` や `map_or` 等を使用し、失敗してもアプリケーション全体がクラッシュしないように保護します。

## 3. コンポーネント・モジュール分割基準

- **単一ファイルの維持**:
  MiSysMon はフットプリントの削減とシンプルさを重視するため、現状は `src/main.rs` のみで構成されています。原則として、新しい小規模な補助機能は `src/main.rs` に定義します。
- **モジュール分割の契機**:
  - 単一のソースファイル（`src/main.rs` 等）のコード行数が **1,000 行**を超過した場合、または主要コンポーネント（UI描画ロジック `eframe::App` 実装やシステムメトリクス取得処理 `SystemMonitor` 内ロジック）が 500 行を超え保守性が低下した場合は、大賢者からユーザーへモジュール分割（例: `src/gui.rs` や `src/monitor.rs` への分割・モジュール化）およびリファクタリングを自主的に提案・推進します。
- **共有ライブラリの活用 (`common_lib`)**:
  - ウィンドウの多重起動防止処理やバイト数表示のフォーマットのように、他のデスクトップアプリケーション（同オーナーの他プロジェクト）と共有できる処理は、本リポジトリに直接書くのではなく、親ディレクトリにあるローカル共有クレート `common_lib` からインポートします。

## 4. AIの回答・出力フォーマット指定

AIエージェントがユーザーに回答を提示する際、以下のフォーマットを遵守してください。

- **解説は最小限に**:
  ユーザーへの余計な説明や、すでにコードやドキュメントを読めばわかる冗長な解説は省いてください。結論、または変更された差分 (diff) を簡潔に提示します。
- **コード変更時の提示形式**:
  - contiguous（一箇所の連続した）変更の場合は、`replace_file_content` を使用するための明確なターゲットコードと置換コードの差分を markdown の diff ブロック等で明示してください。
  - コード全体を再出力するような無駄なテキスト出力は避けてください。
- **ドキュメントの自動更新**:
  ソースコードに変更を加えた場合は、必ず `CHANGELOG.md` を更新し、対応する `SPEC.md` や `ARCHITECTURE.md` があれば同時に追記・修正を行ってください。
